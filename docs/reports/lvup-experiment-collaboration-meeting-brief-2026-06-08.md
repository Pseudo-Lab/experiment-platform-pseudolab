# LVUP 협업 및 실험 연동 회의 공유안 초안

> 팀 내부 참고용 초안이다. LVUP 담당자와 공식 조율이 완료된 기준 문서가 아니며,
> 실제 공유 전에는 책임 경계, 일정, SDK 적용 범위, 리뷰 요청 사항을 다시 확인한다.

## 1. 핵심 메시지

- 현재 이슈: LVUP PR 리뷰 지연으로 인한 12기 회고 실험 일정 지연
- 핵심 원인: 실험마다 LVUP 서비스 코드 변경이 필요한 구조
- 결정된 사항: 12기 회고는 기존 준실험 임시 API 계약으로 진행
- 새 검토 이슈: 12기 회고 일정과 별도로 SDK 개발을 병행하고, SDK가 PR 병목을 줄일 수 있는지 검토
- 중기 방향: 신규 A/B 테스트부터 SDK와 LVUP 실험 슬롯 기반 운영으로 전환
- 최종 목표: 반복 실험은 실험 플랫폼 설정 중심, 새 위치/새 기능은 LVUP PR 중심

## 2. 용어 정리

- LVUP 실험 슬롯: LVUP 화면 안에서 실험용으로 미리 할당한 UI 영역
- 예시: 프로젝트 상세 홈 CTA, 사이드바 공지, 스터디 목록 중간 카드
- 의미: LVUP이 화면 위치와 UI를 소유하고, 실험 플랫폼이 노출 결정값과 payload를 제공하는 연동 지점

## 3. 현재 병목

```text
실험 아이디어
  -> LVUP UI/라우트/이벤트 코드 변경
  -> LVUP PR 작성
  -> LVUP 관리자 리뷰
  -> 머지/배포
  -> 실험 플랫폼 설정
  -> QA
  -> 실험 시작
```

- 문제: 실험 일정이 LVUP PR 리뷰 일정에 종속
- 영향: 실험 시작일 예측 어려움
- 개선 방향: 사전 합의된 LVUP 실험 슬롯 안에서는 실험 플랫폼 설정만으로 반복 실험 가능

## 4. 책임 경계

| 영역 | 실험 플랫폼 | LVUP |
|---|---|---|
| 실험 상태/기간/rollout | 소유 | 참고 |
| 타겟팅/variant 결정 | 소유 | context 제공 |
| SDK/API 계약 | 소유 | 연동 |
| UI 컴포넌트/레이아웃 | payload 제공 | 소유 |
| 라우트/사용자 플로우 | 참고값 제공 | 소유 |
| 인증/권한/사용자 식별 | 전달값 사용 | 소유 |
| 노출/클릭/전환 이벤트 | 수집/분석 | 실제 발생 시점 전송 |
| 배포/롤백 | 실험 상태 제어 | 서비스 배포 |

## 5. 운영 원칙

- 12기 회고: 기존 임시 API 계약으로 진행 확정
- SDK 병행: 12기 회고 일정과 별도로 SDK 개발을 병행
- SDK 적용 가능성: SDK 구현/QA가 PR 병목을 줄이고 운영 리스크가 낮은지 별도 검토
- 신규 A/B 테스트: SDK v1 완성 이후 SDK 표준 연동 전환
- SDK/API v1 확정 전: endpoint, payload schema, event contract 최종 확정 보류
- LVUP 실험 슬롯의 문구/링크/payload 변경: 실험 플랫폼 설정으로 처리
- LVUP 실험 슬롯의 새 UI 타입: LVUP에 사전 구현된 타입만 설정으로 처리
- 새 화면 위치/새 기능/개인정보·권한 영향: LVUP PR과 사전 검토 필요
- 실험 시작일: LVUP PR 필요 여부와 리뷰 lead time 확인 후 확정

## 6. UI 영향도 기준

| 레벨 | 변경 유형 | 예시 | LVUP PR |
|---|---|---|---|
| Level 0 | 분석 이벤트만 추가 | 페이지 진입, 클릭 이벤트 | 필요할 수 있음 |
| Level 1 | LVUP 실험 슬롯 payload 변경 | title, description, target_url | 불필요 |
| Level 2 | LVUP 실험 슬롯 UI 타입 변경 | banner, card, cta | 사전 구현 시 불필요 |
| Level 3 | 새 화면 위치에 슬롯 추가 | 사이드바 CTA | 필요 |
| Level 4 | 새 기능/새 라우트 | 설문 페이지, 온보딩 플로우 | 필요 |
| Level 5 | 개인정보/권한/알림 영향 | 개인정보 수집, 권한 분기 | 필요 + 정책 검토 |

분석가가 영향도를 판정할 때는 해당되는 가장 높은 레벨을 선택한다. PR 필요 여부는 영향도 레벨과 별도로 판단한다. 예를 들어 Level 0 분석 이벤트라도 LVUP 코드에 새 이벤트 전송 위치를 추가해야 하면 PR이 필요할 수 있다.

## 7. 이미 정한 사항과 새 검토 이슈

- 결정된 사항: 12기 회고는 기존 준실험 임시 API 계약으로 진행
- 새 검토 이슈: SDK 개발을 병행하고, SDK가 PR 병목을 줄일 수 있는지 검토
- 신규 A/B 테스트: SDK v1 완성 이후 SDK 표준 연동으로 전환
- LVUP 리뷰 요청: 실험별 템플릿 사용 권장

## 8. 이번 주 내부 정리 항목

### 8-1. SDK 완료 전 지금 정리할 항목

1. LVUP에 제안할 책임 경계 기준
   - 이유: 실험 플랫폼이 어디까지 결정하고, LVUP이 어디부터 UI/라우트/배포를 소유하는지 명확화
   - 산출물: LVUP에 전달할 책임 경계 문구

2. 우선 구성할 LVUP 실험 슬롯 목록
   - 이유: 모든 화면을 한 번에 슬롯화할 수 없으므로 반복 실험 가능성이 높은 위치부터 선정
   - 예시: 프로젝트 상세 홈 CTA, 프로젝트 사이드바 공지, 스터디 목록 중간 카드

3. LVUP PR 필요 실험의 최소 lead time
   - 이유: 실험 시작일을 PR 리뷰 일정과 분리해서 예측 가능하게 관리
   - 예시: 슬롯 설정 변경 1~2일, 새 슬롯 1~2주, 새 기능 2주 이상

4. SDK 병행 개발 및 12기 회고 적용 가능성
   - 이유: 12기 회고 API 진행은 결정됐지만, SDK 개발을 병행하면 이후 반복 실험의 PR 병목을 줄일 수 있음
   - 산출물: SDK 병행 범위, 12기 회고 적용 가능성 판단 기준, fallback 계획
   - 판단 체크:
     - 기존 API와 동일한 decide 결과를 낼 수 있는가
     - viewed/clicked/submitted 이벤트가 누락 없이 기록되는가
     - SDK 실패 시 기존 API 또는 UI hide fallback이 가능한가
     - LVUP 쪽 변경량이 기존 PR보다 줄어드는가
     - SDK 적용이 12기 회고 일정을 지연시키지 않는가

5. LVUP에 공유할 조율안과 요청사항 범위
   - 이유: 내부 기준을 정한 뒤 LVUP에는 구현 가능성, 리뷰 lead time, 배포 방식 중심으로 요청
   - 산출물: LVUP에 전달할 실험 슬롯 후보, 책임 경계 초안, 리뷰 요청 템플릿 초안

6. LVUP 리뷰 요청 템플릿 기본 구조
   - 이유: 템플릿 사용 자체는 확정, 다만 SDK/API 세부 계약은 아직 변경 가능
   - 예시: 영향도 레벨, 슬롯 키, payload schema, 이벤트 계약, QA/롤백 기준

### 8-2. SDK v1 이후 최종 확정할 항목

1. 슬롯별 허용 payload와 UI 타입 범위
   - 이유: SDK/API 응답 구조가 확정된 뒤 payload schema 최종화 필요
   - 현재 단계: 후보 예시만 정리

2. 노출 이벤트 기준과 구현 방식
   - 이유: `decide`, `expose`, `track` SDK 계약 확정 후 이벤트 전송 방식 최종화 필요
   - 현재 단계: 실제 렌더링 또는 viewport 노출 기준 권장

3. LVUP 리뷰 요청 템플릿 주요 입력 항목
   - 이유: SDK/API v1 필드가 확정된 뒤 주요 항목 고정 가능
   - 현재 단계: 영향도, 슬롯, QA/롤백 중심의 임시 구조 사용

## 9. 권장 방향

```text
12기 회고:
  기존 준실험 API 계약으로 진행
  SDK 개발은 병행
  SDK의 12기 회고 적용 가능성은 별도 검토 이슈로 관리

신규 A/B 테스트:
  SDK 표준 연동 준비
  decide / expose / track 계약 정리

LVUP 협업:
  서비스 UI와 slot은 LVUP 소유
  실험 플랫폼은 승인된 slot의 decision/payload/event 설정
  실험별 LVUP 리뷰 요청 템플릿 사용 권장
```
