# Dashboard PM Review (MVP)

작성일: 2026-03-10  
작성자: PM Subagent

## 1) 목적

현재 문서(`dashboard-mvp-plan.md`, `dashboard-test-strategy.md`, Cloudflare D1/R2 조사 문서)를 기준으로 대시보드 MVP의 **요구사항/범위/우선순위**를 점검하고, 개발 완료 판단을 위한 **수용 기준(acceptance criteria)** 및 **리뷰 포맷**을 정의한다.

---

## 2) 요구사항/범위/우선순위 점검

## 2.1 MVP 핵심 요구사항 (정리)

대시보드는 다음 3개 기능 축을 중심으로 구성한다.

1. **KPI 요약 카드**
   - 총 실험 수
   - 활성 실험 수
   - 총 참여자 수
   - 승률/전환율(정의 확정 필요)

2. **시계열 트렌드 차트**
   - 최근 N일 추세
   - 지표는 MVP에서 1개(예: 참여자/전환율)로 시작

3. **랭킹 컴포넌트**
   - 실험별 성과 Top N
   - 상태(draft/active/completed) 표시

추가 공통 요구:
- 로딩/성공/빈 상태/에러 상태를 UI에서 명시적으로 구분
- 다국어(ko/en) 문자열 동기화
- shadcn/ui + 현재 디자인 톤 유지

## 2.2 범위(Scope) 판단

### In Scope (MVP에 포함)
- 프론트 구조 분리(`Dashboard.tsx` + 하위 mvp 컴포넌트)
- 하드코딩 KPI 제거 및 API/파생 데이터 기반 표시
- 상태별 UI 분기(loading/empty/error/success)
- 기본 테스트 보강(P0 우선)

### Out of Scope (MVP 이후)
- 신규 백엔드 분석 파이프라인 대규모 구축
- 복수 지표 동시 비교, 고급 필터, 커스텀 기간 비교
- R2 객체 레벨 분석 대시보드(현 권한/조사 범위상 불확실)
- 완성형 데이터 거버넌스(용어사전 full 구축)

## 2.3 우선순위 점검 (권고)

### P0 (반드시 완료)
1. 구조 분리(유지보수성 확보)
2. KPI 실데이터화(하드코딩 제거)
3. 상태 분기 UI(특히 에러 fallback)
4. 핵심 회귀 테스트 3종(매핑/실패/언어 전파)

### P1 (MVP 완성도)
1. 트렌드 차트 연결(데이터 정의 확정 버전)
2. Top N 랭킹 연결
3. 접근성 기본 점검(heading/role/버튼 이름)

### P2 (다음 스프린트)
1. 지표 사전(glossary_terms) 정비 연계
2. 데이터 품질 모니터링(커버리지/신선도)
3. 고급 탐색 기능(필터, drill-down)

## 2.4 선결 이슈 (결정 필요)

1. **승률/전환율 정의 확정**
   - 분자/분모/집계 기간을 명시하지 않으면 해석 충돌 위험
2. **트렌드 지표의 데이터 소스 확정**
   - 임시 파생인지, 별도 summary endpoint인지 결정 필요
3. **랭킹 score 산식 공개 수준 결정**
   - 사용자에게 노출되는 점수 의미 설명 필요
4. **에러 UX 기준**
   - 토스트/인라인 메시지/재시도 버튼 포함 여부

## 3) Acceptance Criteria 체크리스트

다음 항목을 모두 만족하면 MVP 완료로 간주한다.

## 3.1 기능(Function)
- [ ] `/dashboard` 진입 시 KPI/트렌드/랭킹 3개 섹션이 모두 렌더링된다.
- [ ] 로딩 중에는 progress UI가 표시되고, 완료 후 제거된다.
- [ ] 데이터가 없으면 empty-state가 노출된다.
- [ ] API 실패 시 에러 상태 UI가 노출되고 화면이 크래시되지 않는다.
- [ ] status(`active|draft|completed`) 라벨이 언어별로 정확히 매핑된다.
- [ ] KO/EN 토글 시 대시보드 핵심 텍스트(헤더/상태/empty/error)가 동기화된다.

## 3.2 데이터 정합성(Data Integrity)
- [ ] KPI 값은 하드코딩이 아닌 API 응답/파생 계산 결과를 사용한다.
- [ ] `total_experiments === experiments.length` 규칙이 보장된다.
- [ ] `active_experiments === status==='active'` count와 일치한다.
- [ ] 트렌드 데이터는 날짜 오름차순/중복 없음/결측 처리 규칙이 정의되어 있다.
- [ ] 랭킹 Top N은 동일 기준으로 정렬되며 동점 처리 규칙이 명시된다.
- [ ] 숫자/퍼센트 포맷 규칙(반올림 자리수, locale)이 문서화되어 있다.

## 3.3 UI/UX & 접근성
- [ ] shadcn/ui 컴포넌트 기반으로 구현되고 하드코딩 스타일 남용이 없다.
- [ ] `h1` 포함 heading 구조가 논리적이다.
- [ ] 주요 interactive 요소(언어/테마/사이드바/재시도 등)에 접근 가능한 이름이 있다.
- [ ] 색상/아이콘만으로 상태를 전달하지 않고 텍스트 대체가 존재한다.
- [ ] 모바일/데스크톱에서 레이아웃 깨짐 없이 핵심 정보가 노출된다.

## 3.4 테스트 & 품질
- [ ] Dashboard 단위 테스트에서 loading/success/empty/error 4분기 검증 완료
- [ ] status 매핑 테스트(active/draft/completed) 통과
- [ ] API reject 케이스에서 크래시 방지 테스트 통과
- [ ] App-대시보드 통합 테스트(언어 전파/라우팅) 통과
- [ ] CI 기준 lint/test/typecheck 모두 통과

## 3.5 보안/운영
- [ ] 토큰/비밀키/내부 자격증명 하드코딩이 없다.
- [ ] API endpoint 및 환경변수 참조 방식이 코드 표준을 따른다.
- [ ] 실패 로그는 추적 가능하되 민감정보를 포함하지 않는다.

## 5) PM 결론

현재 문서 기준으로 MVP 방향은 타당하며, 특히 **구조 분리 → 실데이터화 → 상태 분기 명시화 → P0 테스트 강화** 순서가 안전하다.  
다만, 지표 정의(승률/전환율), 트렌드 소스, 랭킹 산식의 제품 정의가 선행되지 않으면 구현 후 해석 불일치가 발생할 가능성이 높다.

따라서 본 리뷰 문서의 Acceptance Criteria를 개발 완료 조건으로 고정하고, 선결 이슈 4가지를 스프린트 초반에 확정할 것을 권고한다.
