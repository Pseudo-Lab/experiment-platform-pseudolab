## 4. 핵심 모듈 계약

### 4.1 `lib/userId.ts`

```ts
export function getUserId(): string {
  let id = localStorage.getItem('demo_uid');
  if (!id) {
    id = crypto.randomUUID();
    localStorage.setItem('demo_uid', id);
  }
  return id;
}

// 개발/시연용: user_id 강제 변경 → 다른 variant 확인 시
export function resetUserId(): string {
  const id = crypto.randomUUID();
  localStorage.setItem('demo_uid', id);
  return id;
}
```

### 4.2 `lib/sdk.ts`

세 함수만 export. 플랫폼 API base URL은 `import.meta.env.VITE_PLATFORM_API` 사용.

```ts
const API = import.meta.env.VITE_PLATFORM_API; // 예: http://localhost:8000/api/v1

export async function decideFlag(flagKey: string): Promise<string> {
  const uid = getUserId();
  const res = await fetch(
    `${API}/feature-flags/decide?flag_key=${flagKey}&user_id=${uid}`
  );
  const json = await res.json();
  return json.data.variant; // 'control' | 'treatment' | 커스텀
}

export async function track(
  eventName: string,
  properties?: Record<string, unknown>
): Promise<void> {
  await fetch(`${API}/capture`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      user_id: getUserId(),
      event_name: eventName,
      properties,
    }),
  });
}

export async function identify(meta: {
  cohort_id?: string;
  team_name?: string;
  role?: string;
}): Promise<void> {
  await fetch(`${API}/identify`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ user_id: getUserId(), ...meta }),
  });
}
```

> ⚠️ **identify는 upsert이며 누락 필드를 NULL로 덮어씀** (선행 문서 §7). 부분 업데이트 시 기존값을 먼저 읽거나, 항상 모든 필드를 함께 보내야 함. 1차 구현은 호출 자체를 최소화(예: 첫 진입 시 1회) 권장.

### 4.3 `hooks/useFlag.ts`

sessionStorage 캐시 포함. 캐시 키는 `flag:${flagKey}:${userId}` — user_id 변경 시 자동 재요청.

```ts
export function useFlag(flagKey: string): string | null {
  const [variant, setVariant] = useState<string | null>(null);

  useEffect(() => {
    const uid = getUserId();
    const cacheKey = `flag:${flagKey}:${uid}`;
    const cached = sessionStorage.getItem(cacheKey);
    if (cached) {
      setVariant(cached);
      return;
    }
    decideFlag(flagKey).then((v) => {
      sessionStorage.setItem(cacheKey, v);
      setVariant(v);
    });
  }, [flagKey]);

  return variant;
}
```

**렌더 규칙**: `variant === null`인 동안은 skeleton/placeholder 표시 (FOUC 방지). 분기 변경 깜빡임이 시연 시 가장 거슬리는 포인트.

## 5. 페이지별 구현 가이드

### 5.1 HomePage (`/`) — B-1

```tsx
function HomePage() {
  const variant = useFlag('home_layout_v1');
  if (!variant) return <StudySkeleton />;

  const studies = mockStudies;
  const onCardClick = (study: Study) =>
    track('study_card_clicked', { variant, study_id: study.id });

  return variant === 'list'
    ? <StudyList studies={studies} onClick={onCardClick} />
    : <StudyGrid studies={studies} onClick={onCardClick} />;
}
```

- `StudyGrid`: Tailwind `grid grid-cols-3 gap-4`
- `StudyList`: Tailwind `flex flex-col gap-3`
- 두 컴포넌트 모두 내부에서 `<StudyCard>` 재사용, prop으로 `layout: 'grid'|'list'`만 다름

### 5.2 StudyListPage (`/studies`) — A-1

```tsx
function StudyListPage() {
  const variant = useFlag('sponsor_slot_v1');
  if (!variant) return <PageSkeleton />;

  const onSponsorClick = (sponsor: Sponsor) =>
    track('sponsor_clicked', { variant, sponsor_id: sponsor.id });

  // sidebar: 본문 + 우측 사이드바에 SponsorBanner
  // inline:  스터디 카드 3개마다 SponsorBanner를 카드 자리에 삽입
  return (
    <div className={variant === 'sidebar' ? 'flex gap-6' : ''}>
      <main>{/* 스터디 목록 + (inline이면 중간중간 배너) */}</main>
      {variant === 'sidebar' && <aside><SponsorBanner onClick={onSponsorClick} /></aside>}
    </div>
  );
}
```

> **노출 이벤트는 1차 범위 외**. 클릭 이벤트만 잡음. 노출 대비 CTR이 필요해지면 2차 작업에서 `sponsor_impressed` 추가하고 IntersectionObserver로 1회만 보냄.

## 6. docker-compose 통합

`docker-compose.dev.yml`에 서비스 추가:

```yaml
  experiment-platform-pseudolab-demo-app:
    build:
      context: ./examples/demo-app
      dockerfile: Dockerfile
    container_name: experiment-platform-pseudolab-demo-app-dev
    ports:
      - "8082:5173"
    volumes:
      - ./examples/demo-app:/app
      - /app/node_modules
    command: npm run dev -- --host 0.0.0.0
    restart: always
    environment:
      - VITE_PLATFORM_API=http://localhost:8000/api/v1
```

`Dockerfile`은 `frontend/Dockerfile`을 참고해 동일 패턴(dev target)으로 작성.

`docker compose -f docker-compose.dev.yml up` 한 번으로:
- backend: `http://localhost:8000`
- 관리자 콘솔: `http://localhost:8081`
- demo-app: `http://localhost:8082`

## 7. 작업 순서 (Step-by-step)

각 단계 종료 시점에 동작 확인 후 다음 단계로 진행.

### Step 1 — 관리자 콘솔 사전 준비 (코드 변경 없음)

> ⛔ **이 단계를 건너뛰면 SDK는 멀쩡히 작동해 보입니다.** 백엔드 `decide`는 DB에 없는 flag에 대해 조용히 `"control"`을 반환하고(`reason: "unknown_flag"`, [`backend/app/services/feature_flag.py:263-266`](../../backend/app/services/feature_flag.py)) 노출 기록도 생략합니다([`backend/app/services/feature_flag.py:199-203`](../../backend/app/services/feature_flag.py)). 즉 데모 화면은 정상 동작하지만 모든 사용자가 같은 variant로 떨어지고, 콘솔의 노출 카운트는 0인 상태가 됩니다. 에러가 없어 디버깅이 어려우니 **반드시 Step 2 코드 작업 전에 끝낼 것.**
>
> 자동화 옵션은 `20260517_issues.md` 이슈 3에 정리된 `examples/demo-app/scripts/seed.sh` 권장.

## 9. 1차에서 의도적으로 제외한 것 / 주의 사항

### 9.1 컨벤션 격리 (의도적 제외)

루트 `AGENTS.md`의 아래 규칙은 1차에서 미적용:

- **shadcn/ui & 디자인 시스템** (`AGENTS.md §4`) — demo-app은 "외부 SDK 사용자가 보는 앱"의 포지션이므로 관리자 콘솔과 디자인 격리. 1차는 plain Tailwind, 시연/공유 단계에서 도입 검토.
- **KO/EN i18n** (`AGENTS.md §5`) — 1차는 KO 단일. 데모 시나리오에 광고 카피 다국어(A-2, 선행 문서 §2)가 들어가면 그때 도입.

> 이 결정은 사용자(seongdong)와 합의된 사항이며, 본 데모는 관리자 콘솔과 분리된 "예시 앱" 위치라서 격리한 것. 본 플랫폼 작업에는 AGENTS.md 규칙이 그대로 유지됨.

### 9.2 향후 작업으로 미룬 것 (선행 문서 §4·§5 대비)

- 세그먼트 타게팅, 시간 윈도우, 점진 rollout (5%→50%→100%)
- cohort 메타데이터 기반 분해 분석 (선행 문서 §4)
- 스터디 상세 / 지원 폼 / 완료 페이지 (퍼널 깊이)
- 노출 이벤트 (`sponsor_impressed`, `study_card_impressed`)
- 부정 지표 (`session_bounced` 등, 선행 문서 §2 A-3)

### 9.3 플랫폼 측 주의 사항 (선행 문서 §7 재확인)

- **CORS는 backend가 `["*"]`** — 로컬 OK, 운영 전환 시 origin 명시 필요
- **D1 환경변수 필수** — `CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_API_TOKEN`, `D1_DATABASE_ID` 누락 시 모든 쿼리가 조용히 빈 결과 반환 → 콘솔에 데이터가 안 보일 때 첫 의심 지점
- **decide API에 캐싱 없음** — 본 데모는 `useFlag` 훅에서 sessionStorage 캐시로 대응
