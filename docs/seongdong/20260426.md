## 1. 실험 플랫폼 관점 — 무엇이 추가됐는가

### 1-1. 실험 결과 자동 계산 (Bayesian A/B Result)

실험이 완료된 후 `/experiments/{id}/result` 를 호출하면 아래 지표를 자동 계산한다.

| 지표 | 설명 |
|------|------|
| `rate` | 각 Variant의 전환율 (conversions / users) |
| `uplift` | treatment rate − control rate |
| `probability_treatment_wins` | Bayesian 확률. treatment가 control보다 좋을 확률 (0~1) |
| `srm_warning` | Sample Ratio Mismatch 경고 여부 (true/false) |
| `sample_size` | 분석에 사용된 전체 유저 수 |

**계산 방식**

```
Beta(α = conversions + 1, β = non-conversions + 1) 사후 분포에서
10,000회 Monte Carlo 샘플링 → P(treatment > control) 계산

SRM 검정: Chi-squared test (p < 0.01 이면 경고)
```

- "전환"은 `event_log` 테이블에서 `event_name = primary_metric` 인 이벤트 발생 여부로 판단한다.
- `primary_metric` 은 실험 생성 또는 수정 시 설정한다.
- Variant 배정은 기존 SHA-256 해시 기반 결정론적 배정(`experiment_assignments` 테이블)을 그대로 사용한다.

### 1-2. 이벤트 수집 (Event Log)

분석 지표 계산과 A/B 전환율 측정의 원천 데이터다.

```
POST /capture
{
  "user_id": "u-001",
  "event_name": "button_click",    ← primary_metric 과 일치해야 전환으로 인식
  "properties": {"page": "home"}   ← 선택
}
```

- `event_log` 테이블에 적재된다.
- `person` 테이블에 등록된 사용자는 `cohort_id` 가 자동 연결된다.

**사용자 식별 (Identify)**

```
POST /identify
{
  "user_id": "u-001",
  "cohort_id": "cohort-2024-spring",
  "cohort_name": "2024 봄 기수",
  "team_name": "팀A",
  "role": "researcher"
}
```

- `person` 테이블 UPSERT. 기존 레코드가 있으면 덮어쓴다.
- 이후 해당 사용자의 이벤트에는 cohort 정보가 자동 태깅된다.

### 1-3. Feature Flag

트래픽의 일부에게만 기능을 노출할 때 사용한다. A/B 실험과 독립적으로 운영된다.

| 필드 | 설명 |
|------|------|
| `flag_key` | 고유 식별자 (예: `dark-mode-v2`) |
| `rollout_pct` | 0~100. 해당 비율의 유저에게 활성화 |
| `enabled` | false 이면 rollout_pct 무관하게 전체 비활성 |

**배정 방식**: `CRC32(user_id:flag_key) % 100 < rollout_pct`

```
GET /feature-flags/decide?flag_key=dark-mode-v2&user_id=u-001
→ { "enabled": true }
```

### 1-4. 분석 대시보드 지원 API

`/analytics` 하위에 3가지 분석 쿼리를 제공한다.

**트렌드 (Trends)**
- 특정 이벤트의 일별 / 주별 발생 건수 추이
- `GET /analytics/trends?event_name=button_click&start=2026-04-01&end=2026-04-26&granularity=day`

**퍼널 (Funnels)**
- 이벤트 시퀀스를 순서대로 완료한 유저 수
- `POST /analytics/funnels` — `{ "steps": ["signup", "onboarding", "purchase"], "start": ..., "end": ... }`

**리텐션 (Retention)**
- 첫 이벤트 이후 N주차에 재방문한 유저 비율 (최대 12주)
- `GET /analytics/retention?event_name=login&start=...&end=...`

### 1-5. 의사결정 이력 (Decision Log)

실험 완료 후 팀이 내린 결정을 INSERT-ONLY로 기록한다. 삭제·수정 불가.

```
POST /decisions
{
  "experiment_id": "exp-001",
  "decision": "SHIP",        ← SHIP / HOLD / ROLLBACK 중 하나
  "reason": "uplift 8%, p > 0.95",
  "decided_by": "seongdong"
}
```

**학습 노트 (Learning Note)**: 실험에서 얻은 인사이트를 자유 형식으로 기록한다.

```
POST /learning-notes
{
  "experiment_id": "exp-001",
  "content": "모바일 유저에서만 uplift가 유의미했음",
  "created_by": "seongdong"
}
```

### 1-6. 회고 (Reflection)

실험 참여자가 프로젝트 회고를 제출하는 기능. **1인 1실험 1회**만 제출 가능.

| 필드 | 설명 |
|------|------|
| `project_type` | research / implementation / presentation / document / opensource |
| `response_good` | 잘 된 점 |
| `response_blocked` | 막혔던 점 |
| `response_goal` | 다음 목표 |

```
GET /reflections/check?experiment_id=exp-001&user_id=u-001
→ { "submitted": false }   ← 제출 여부 확인

GET /reflections/summary?experiment_id=exp-001
→ 프로젝트 유형별 집계 및 전체 답변 목록
```

| 파일 | 내용 |
|------|------|
| `005_event_collection_feature_flags.sql` | `event_log`, `person`, `feature_flag`, `cohort` 테이블 생성 |
| `006_experiment_extensions_decision_log.sql` | `experiments` 테이블 컬럼 추가, `decision_log`, `learning_note` 테이블 생성 |

**적용 순서:**
```bash

bash backend/migrations/run_migration.sh 005
bash backend/migrations/run_migration.sh 006

```

필요한 환경변수:
```

CLOUDFLARE_API_TOKEN=<D1 편집 권한 포함 API Token>
D1_DATABASE_ID=<pseudolab-exp DB ID>
```

### 2-2. 실험 결과 계산 로직 (`experiment.py`)

```python
# Bayesian: Beta 사후 분포 기반
t_samples = np.random.beta(t_success + 1, (t_total - t_success) + 1, 10_000)
c_samples = np.random.beta(c_success + 1, (c_total - c_success) + 1, 10_000)
prob = float((t_samples > c_samples).mean())

# SRM: Chi-squared test
_, p_value, _, _ = chi2_contingency(
    [[t_total, c_total], [t_total + c_total, t_total + c_total]]
)
srm_warning = bool(p_value < 0.01)
```

- Variant 이름 "control" / "treatment" 를 기준으로 그룹을 구분한다.
- 해당 이름이 없으면 첫 번째 / 두 번째 Variant로 대체한다.

### 2-3. Analytics SQL — SQLite 적응

Cloudflare D1은 SQLite 기반이라 MySQL 문법을 사용할 수 없다.

| MySQL | D1/SQLite 대체 |
|-------|---------------|
| `YEARWEEK(dt)` | `strftime('%Y-%W', dt)` |
| `DATEDIFF(a, b)` | `CAST(JULIANDAY(a) - JULIANDAY(b) AS INTEGER)` |
| `DATE_TRUNC('week', dt)` | `strftime('%Y-%W', dt)` |

### 2-4. Feature Flag 배정 알고리즘

```python
import binascii
bucket = binascii.crc32(f"{user_id}:{flag_key}".encode()) % 100
return enabled and bucket < rollout_pct
```

- SHA-256 대비 연산 속도가 빠르다.
- 결정론적이므로 같은 (user_id, flag_key) 조합은 항상 동일한 결과를 반환한다.

### 2-5. 실험 생성 수정 이력 (버그 수정)

신규 컬럼(`expected_effect`, `primary_metric`, `cohort_id`)이 마이그레이션 006 적용 전에 INSERT에 포함되어 D1에서 404 → `d1.execute()` → `False` → `get()` → `None` → FastAPI ResponseValidationError 가 발생했다.

수정: INSERT 구문을 기존 컬럼만 포함하도록 롤백. 신규 컬럼은 마이그레이션 006 적용 후 사용 가능.

### 3-2. 모달 오버레이 버그 수정 (`CreateExperimentModal.tsx`)

**문제**: `<main>` 태그에 `transition-all duration-300` CSS가 적용돼 있어, 브라우저가 별도 합성 레이어(compositing layer)를 생성한다. 이 안에 `position: fixed` 요소를 두면 뷰포트가 아닌 `<main>` 기준으로 위치가 계산되어, sticky header(z-30) 위로 올라오지 못했다.

**해결**: `createPortal(…, document.body)` 로 모달을 `<body>` 직속 자식으로 렌더링하고 `z-[9999]` 적용.

```tsx
import { createPortal } from 'react-dom';

return createPortal(
  <div className="fixed inset-0 z-[9999] flex items-center justify-center bg-black/50">
    {/* 모달 내용 */}
  </div>,
  document.body
);
```
