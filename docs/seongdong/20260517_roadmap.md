## 2. 구간 0 — Blocker (이슈 5 해결)

**왜 먼저**: 이 문제를 안 풀고 추가 시나리오를 늘리면 모든 후속 실험의 데이터가 무의미. 측정 정확도 보강(구간 1)도 무의미한 데이터에 정밀도만 더하는 꼴.

**작업**:
1. issues.md §이슈5의 A/B/C 안 중 하나 결정 (요구되는 산출물: 결정 노트 1쪽)
2. 선택된 안으로 백엔드 또는 SDK 수정
3. 검증 (issues.md §이슈5 검증 방법 5단계)

**완료 조건**: 동일 user_id 100명 시뮬레이션에서 Flag variant ↔ Experiment variant 일치율 ≥ 95%.

**참고**: B(assign이 Flag decide 결과 재사용)가 의미적으로 가장 옳지만 변경 폭이 큼. 단기적으로 C(데모 SDK가 assign 결과로 분기) + 장기적으로 B 마이그레이션이 현실적 경로.

## 3. 구간 1 — 단기 (측정 정확도)

### 3-1. 노출 이벤트 도입

**무엇**: `study_card_impressed`, `sponsor_impressed` 이벤트를 IntersectionObserver로 1회만 발생. CTR = `clicked / impressed` 분모 정확화.

**왜**: 지금은 "본 사람" 분모가 없어서 단순 절대 클릭 수만 비교. 스크롤 안 한 사용자가 분모에 포함되면 위치/디자인 효과가 왜곡됨.

**스코프**:
- `examples/demo-app/src/hooks/useImpression.ts` 추가 (IntersectionObserver 래핑)
- `StudyCard.tsx`, `SponsorBanner.tsx`에 훅 적용
- 실험의 primary_metric은 그대로 `*_clicked` 유지하되, 콘솔에서 CTR 계산식 추가 (또는 secondary_metric으로 `*_impressed` 등록)

**놓치기 쉬운 점**: 같은 사용자가 같은 요소를 여러 번 봐도 1회만 발생해야 함. sessionStorage 또는 컴포넌트 unmount 시 dedupe.

### 3-2. 퍼널 1단 확장 (스터디 상세 페이지)

**무엇**: `/studies/:id` 상세 페이지 추가 + `study_detail_viewed`, `study_apply_clicked` 이벤트.

**왜**: 클릭률 차이만으론 "list가 진짜 더 좋은 사용자 경험인가" 알 수 없음. 클릭 → 상세 → 지원 시작까지 도달률을 봐야 의미 있음.

**스코프**:
- `pages/StudyDetailPage.tsx` 추가
- react-router에 동적 라우트 등록
- HomePage / StudyListPage의 카드 클릭 → `navigate(/studies/:id)`
- 콘솔 `/api/v1/analytics/funnels` 활용 가능성 확인 (선행 문서 §2 B-2 참고)

**시나리오 후보 연결**: 선행 문서 §2-B-2 (지원서 폼 길이)로 자연스럽게 연결됨.

## 4. 구간 2 — 중기 (분석 깊이)

### 4-1. cohort 메타데이터 + identify 연동

**무엇**: demo-app 진입 시 `identify({ cohort_id, role })` 1회 호출. 가짜 cohort 3~5개 더미 데이터로 시작.

**왜**: 선행 문서 §4의 "PseudoLab/LuvP 광고 가치 측정" 비전 구현 시작점. 단순 전체 CTR이 아닌 "Builder cohort X%, Student Y%" 분해 가능.

**놓치기 쉬운 점**: identify는 upsert이며 누락 필드를 NULL로 덮어씀(`20260517.md` §7). 부분 업데이트 금지.

### 4-2. 세그먼트별 결과 분해

**무엇**: 콘솔 실험 결과 화면에 cohort/role/team_name 등 메타 차원으로 결과를 break down.

**스코프**: 백엔드 분석 쿼리 + 콘솔 UI. 백엔드는 `event_log.user_id`로 `user_profile` JOIN. 콘솔은 차원 선택 드롭다운 + 결과 테이블.

**전제**: 4-1 완료 (cohort 데이터 쌓여야 분해 가능).

### 4-3. 가드레일 메트릭

**무엇**: 실험에 `guardrail_metrics: ['session_bounced', 'error_rate']` 같은 부정 지표 등록. treatment의 primary는 올랐지만 guardrail이 악화되면 콘솔에서 경고.

**왜**: 진짜 의사결정엔 "이겼다"보다 "이겼고 부작용 없다"가 중요.

## 5. 구간 3 — 장기 (LuvP 본격 도입)

### 5-1. 점진 rollout (canary) 워크플로

**무엇**: 5% → 25% → 50% → 100% 단계적 rollout. 각 단계에서 가드레일 정상이면 자동 다음 단계, 악화 시 자동 롤백.

**스코프**: 백엔드 스케줄러 + 콘솔 UI(현재 rollout %, 다음 단계 예정 시각, 자동/수동 토글).

### 5-2. 실험 자동 종료

**무엇**: P > 0.95 || (샘플 사이즈 N 도달 && 차이 미검출) 시 자동 종료. 종료 사유 기록.

**전제**: 사전 power analysis로 N 계산이 가능해야 함. 별도 도구 또는 콘솔 내 계산기.

### 5-3. 결정 기록

**무엇**: 실험 종료 후 "채택/기각/연장" + 이유를 콘솔에 기록. 향후 비슷한 시나리오 진행 시 참고.

### 5-4. LuvP 실 트래픽 SDK 도입

**무엇**: demo-app의 SDK(50줄)를 LuvP 실제 앱에 이식. 환경 변수로 dev/prod 분리.

**선결 조건**: 구간 0/1/2 완료. 특히 이슈 5 미해결 상태로 실 트래픽 도입하면 모든 의사결정이 잘못됨.

**놓치기 쉬운 점**: CORS는 backend가 현재 `["*"]` (선행 문서 §7). LuvP 도메인 origin 명시 필요.

## 7. 새 시나리오 추가 워크플로

새 시나리오 추가 시 매번 따르는 절차. 다음 AI/사람이 헷갈리지 않도록 표준화.

| Step | 할 일 | 산출물 |
|---|---|---|
| 1 | 선행 문서(`20260517.md`)에 시나리오 한 단락 추가 | 가설 / variants / primary_metric / 세그먼트 / 시간 윈도우 |
| 2 | `examples/demo-app/scripts/seed.sh` 확장 | Flag + Rule + 실험 idempotent 호출 추가. 기존 실험 PATCH 처리도 포함 권장 |
| 3 | demo-app에 페이지/컴포넌트 추가 또는 기존 페이지 확장 | `useFlag('새_flag_key')` + `track('새_이벤트')` |
| 4 | `examples/demo-app/src/lib/flagToExperiment.ts`에 매핑 추가 | DevPanel이 자동 진단할 수 있음 |
| 5 | seed.sh 재실행 → 시크릿 창 검증 → 콘솔에서 노출 요약·실험 결과 확인 | 검증 스크린샷 또는 로그 |

**주의**: 이슈 5가 해결되기 전엔 새 시나리오 추가의 가치가 낮음. 구간 0 완료 후 진행 권장.
