## 0. 현재 상태 요약

`examples/demo-app/` 1차 구현은 완료됨. docker-compose로 3개 서비스(backend 8000, 관리자 콘솔 8081, demo-app 8082) 정상 기동. 데모 사이트에서 분기 동작·이벤트 수집·노출 누적까지 백엔드에 정확히 저장되는 것 확인됨.

검증 시점(2026-05-17) 백엔드 데이터:

```
GET /api/v1/feature-flags/home_layout_v1/exposure-summary
→ variant_counts: { control: 104, list: 101 }   (총 210 노출, 205 유니크)

GET /api/v1/feature-flags/sponsor_slot_v1/exposure-summary
→ variant_counts: { control: 48, sidebar: 58 }  (총 112 노출, 106 유니크)
```

**데이터 손실은 없음**. 아래 이슈는 모두 "표시"와 "후속 작업" 영역.

### 이슈 1 — 콘솔이 커스텀 variant 카운트를 표시 못 함

**증상**: 콘솔 Feature Flags 페이지에서 `treatment 0 / control 104`로 표기. `list`, `sidebar` 카운트는 화면에 없음.

**원인**: [`frontend/src/features/dashboard/components/FeatureFlags.tsx:239-240`](../../frontend/src/features/dashboard/components/FeatureFlags.tsx)
```ts
const treatment = summary.variant_counts.treatment ?? 0;
const control = summary.variant_counts.control ?? 0;
```
`treatment`라는 고정 키만 읽음 → Rule이 만든 `list`, `sidebar` 등 커스텀 variant 이름은 0으로 누락.

**수정 방향**:
- 같은 파일 [line 248-251](../../frontend/src/features/dashboard/components/FeatureFlags.tsx) 의 렌더 블록을 `Object.entries(summary.variant_counts).map(...)` 로 일반화
- 각 키별 칩(chip)으로 렌더. `control`은 회색, 그 외는 indigo 톤 같은 식으로 시각 구분
- i18n 라벨(`tr.treatment`, `tr.control`)은 키 매칭되면 사용, 아니면 키 그대로 표시
- 빈 카운트(0)는 표시 생략 가능

**스코프**: 프론트엔드 1파일. 백엔드 변경 불필요.
**테스트**: [`frontend/src/__tests__/FeatureFlags.test.tsx`](../../frontend/src/__tests__/FeatureFlags.test.tsx) 에 variant_counts 표시 케이스 추가.
**i18n**: KO/EN 동시 반영 (AGENTS.md §5).

### 이슈 2 — 실험(experiment) 리소스가 미생성

**증상**: 설계서(`20260517_arch.md` §2)에서 약속한 `home_layout_exp_v1`, `sponsor_slot_exp_v1` 실험이 만들어지지 않음. 결과적으로 콘솔 "실험 관리"에서 P(treatment>control), uplift, SRM 같은 통계 결과를 볼 수 없음.

**원인**: Feature Flag와 실험은 별개 리소스.
- Flag: "누구에게 어떤 variant를 보여줄지" 결정 (decide API + 노출 기록)
- 실험: "그 결과로 어느 variant가 더 좋았는지" 분석 (assignments × event_log JOIN, Beta 샘플링 1만 회, 카이제곱 SRM)

1차 구현이 Flag만 생성하고 실험은 빠뜨림.

**필요 작업**:
1. 실험 2개 생성 — 콘솔 "실험 관리" 또는 `POST /api/v1/experiments/`
2. `primary_metric` 매핑:
   - `home_layout_exp_v1` → `study_card_clicked`
   - `sponsor_slot_exp_v1` → `sponsor_clicked`
3. variants는 커스텀 이름 그대로 두 개씩 (`control` + `list`, `control` + `sidebar`), traffic_ratio 0.5/0.5
4. status를 `running`으로 전환해야 분석 대상이 됨

**참고 코드**: [`backend/app/services/`](../../backend/app/services/) 의 experiments 관련 서비스 파일과 스키마. 기존 실험 예시는 `GET /api/v1/experiments/` 응답에서 `382b0495-d7d2-4aca-b802-de882560d461` ("테스트 실험 2026-04", status=running) 참고.

**주의**: 실험 variants 이름이 Flag/Rule에서 만든 variant 이름(`control`, `list`, `sidebar`)과 **정확히 일치**해야 assignments와 매칭됨.

### 이슈 3 — 설계서 Step 1(Flag/실험 사전 생성)이 사람의 수동 작업

**증상**: 설계서 §7 Step 1("관리자 콘솔에서 Flag/실험 사전 생성")을 1차 구현 AI가 빼먹어서, 데모 사이트 첫 띄움 시 모든 사용자가 fallback `control`만 받는 상태로 약 1시간 디버깅 소요.

**근본 원인**: Step 1이 "코드 작업이 아닌 단계"라 구현 AI가 무의식적으로 건너뜀. 게다가 Flag/실험이 없을 때 decide API가 조용히 `control`을 반환하고 노출 기록도 안 남기기 때문에, 데모 사이트는 "동작하는 것처럼" 보이고 문제가 늦게 발견됨.

**해결 방향 후보**:

| 안 | 내용 | 장점 | 단점 |
|---|---|---|---|
| (a) | `examples/demo-app/scripts/seed.sh` — curl로 Flag/Rule/실험을 멱등하게 생성하는 스크립트. `docker compose up` 후 1번 실행 | 가장 단순. 데모 코드 옆에 있어 다음 AI가 발견하기 쉬움 | 사람이 명시적으로 실행해야 함 |
| (b) | backend에 dev-only seed 엔드포인트 (`POST /api/v1/dev/seed-demo`). 데모 환경 변수 켜졌을 때만 동작 | demo-app 첫 로드 시 자동 호출 가능 | 백엔드에 데모 전용 코드 추가, 운영 분리 신경 써야 함 |
| (c) | 설계서 Step 1을 더 강조 + 체크리스트화 | 코드 변경 없음 | 다음 AI도 또 놓칠 가능성 있음 |

**권장**: (a). `scripts/seed.sh` 내부는 다음 5개 호출.

```bash
# 1) Flag 2개 (rollout_pct=50, enabled=true)
POST /api/v1/feature-flags/  { flag_key: 'home_layout_v1',  rollout_pct: 50, enabled: true }
POST /api/v1/feature-flags/  { flag_key: 'sponsor_slot_v1', rollout_pct: 50, enabled: true }

# 2) Rule 2개 (커스텀 variant 이름 매핑)
POST /api/v1/feature-flags/home_layout_v1/rules   { priority: 1, rollout_pct: 50, variant: 'list',    enabled: true }
POST /api/v1/feature-flags/sponsor_slot_v1/rules  { priority: 1, rollout_pct: 50, variant: 'sidebar', enabled: true }

# 3) 실험 2개 (이슈 2 해결)
POST /api/v1/experiments/  {
  name: 'home_layout_exp_v1',
  primary_metric: 'study_card_clicked',
  variants: [
    { name: 'control', traffic_ratio: 0.5 },
    { name: 'list',    traffic_ratio: 0.5 },
  ],
  status: 'running',
}
POST /api/v1/experiments/  {
  name: 'sponsor_slot_exp_v1',
  primary_metric: 'sponsor_clicked',
  variants: [
    { name: 'control', traffic_ratio: 0.5 },
    { name: 'sidebar', traffic_ratio: 0.5 },
  ],
  status: 'running',
}
```

각 호출은 `409 Conflict`(이미 존재) 응답을 정상으로 처리(idempotent). README에 "처음 한 번 실행" 사용법 명시.

### 이슈 4 — DevPanel Experiments 탭 카운트 미스매치

**증상**: DevPanel의 `Experiments (N)` 탭 라벨에 표시되는 N과, 본문에 실제로 렌더되는 실험 카드 개수가 다름. 예: 본문에 `home_layout_exp_v1` + `sponsor_slot_exp_v1` 2개 카드가 표시되는데 탭에는 `Experiments (1)`로 표기.

**원인**: [`examples/demo-app/src/components/DevPanel.tsx:137`](../../examples/demo-app/src/components/DevPanel.tsx)
```ts
<TabBtn label={`Experiments (${Object.keys(assignState).length})`} ... />
```
탭 카운트는 `assignState`(실제 배정 성공한 실험만)을 셈. 반면 본문 [`ExperimentsTab` (line 213-218)](../../examples/demo-app/src/components/DevPanel.tsx)은 `mappedExperiments`(FLAG_TO_EXPERIMENT 매핑된 실험 + 배정된 실험의 합집합)을 표시.

→ 매핑은 됐지만 paused 등으로 배정 실패한 실험은 본문에 "미배정"으로 진단 카드가 뜨지만 탭 카운트에선 빠짐.

**수정 방향**: `mappedExperiments` 계산을 `DevPanel` 부모 컴포넌트로 끌어올려 `useMemo`로 계산 후, 탭 라벨과 `ExperimentsTab`에 함께 전달.

```ts
// DevPanel 내부
const mappedExperiments = useMemo(() => {
  const set = new Set<string>()
  Object.keys(flagState).forEach((flagKey) => {
    const expName = FLAG_TO_EXPERIMENT[flagKey]
    if (expName) set.add(expName)
  })
  Object.keys(assignState).forEach((n) => set.add(n))
  return set
}, [flagState, assignState])

// 탭
<TabBtn label={`Experiments (${mappedExperiments.size})`} ... />

// ExperimentsTab은 prop으로 받아 사용
```

**스코프**: `DevPanel.tsx` 1파일. 테스트 불필요(데모용 진단 패널).

### 이슈 5 — Flag decide와 Experiment assign이 다른 해시 함수 (🔴 critical)

**증상**: 같은 user_id에 대해 Feature Flag와 Experiment가 **무관한** variant를 반환. 예: `home_layout_v1` decide는 `control`을 주는데, `home_layout_exp_v1` assign은 같은 사용자에게 `list`를 배정. 데모 UI는 `control`(grid)을 그리지만 실험 분석 테이블엔 `list`로 기록.

**영향**: AB테스트 플랫폼의 핵심 가치 무력화. 콘솔의 실험 결과 화면(P, uplift, SRM)에 숫자가 나와도, 그 숫자는 "분류된 그룹과 실제로 본 UI가 무관한 데이터"라 **해석 불가**. demo-app뿐 아니라 외부 SDK 사용자(LuvP 등)가 플랫폼을 운영용으로 쓰는 순간 모든 실험 결과가 신뢰 불가.

**원인**: 두 시스템이 독립적인 해시 함수를 사용.

| 시스템 | 호출 | 해시 입력 | 해시 함수 | 위치 |
|---|---|---|---|---|
| Feature Flag | `decide` | `user_id + flag_key + rule_id` | **CRC32** | [`backend/app/services/feature_flag.py:17`](../../backend/app/services/feature_flag.py) |
| Experiment | `assign` | `experiment_id + user_id` | **SHA256** | [`backend/app/services/experiment.py:150`](../../backend/app/services/experiment.py) |

**해결 방향 후보**:

| 안 | 변경 위치 | 장점 | 단점 |
|---|---|---|---|
| A. 백엔드 해시 통일 | `experiment.py:150`을 CRC32 + `flag_key` 기반으로 변경. Experiment 스키마에 `flag_key` 매핑 필드 추가 | 한 곳 수정. 의미적으로 옳음 | 기존 assignments 분포 깨짐. 마이그레이션 필요 |
| B. assign이 Flag decide 결과 재사용 | 백엔드 `assign` 호출 시 매핑된 Flag의 `decide`를 내부 호출. Flag↔Experiment 매핑 테이블 신설 | 가장 깔끔. Flag가 단일 진실 소스 | DB 스키마 + 로직 변경 폭이 큼. 매핑 안 된 실험은 어떻게 할지 정책 필요 |
| C. demo-app 임시 회피 | demo-app이 UI 분기에 Flag 대신 Experiment assign 결과 사용 | 백엔드 무변경 | 플랫폼 근본 문제 잔존. Flag의 rollout/세그먼트/시간 윈도우 기능 못 씀. 다른 SDK 사용자가 또 밟음 |

**권장**: 단기 데모 운영 한정으로 C, 장기적으로는 B. A는 데이터 손실 위험.

**검증 방법**:
1. seed.sh 재실행으로 Flag/Rule/실험 동기화 상태에서
2. 동일 user_id로 100명 시뮬레이션:
   - Flag decide 100회, Experiment assign 100회
   - "Flag에서 control + Experiment에서 control" 사용자 비율이 ~50% 가까이 나와야 정상 (현재는 ~25%로 거의 무관)
3. 콘솔 실험 결과 화면의 conversion rate가 노출 요약의 variant 분포와 정합 확인

**스코프**: 백엔드(A/B) 또는 데모 SDK(C). 어느 안이든 결정 후 별도 설계 문서 필요.

## 3. 다른 AI에 넘길 때 컨텍스트

**반드시 읽어야 할 문서**:
1. [`20260517.md`](./20260517.md) — 시나리오, 플랫폼 API 표면, 주의 사항
2. [`20260517_arch.md`](./20260517_arch.md) — demo-app 구현 설계, 디렉토리/파일 계약
3. 본 문서

**놓치기 쉬운 점**:
- 플랫폼의 기본 `decide`는 `control`/`treatment` 두 값만 반환. **커스텀 variant 이름을 쓰려면 Rule 필수** (서비스 코드 [`backend/app/services/feature_flag.py:21-24`](../../backend/app/services/feature_flag.py))
- Flag/실험이 없으면 `decide`는 **조용히 `control` 반환** (`reason == "unknown_flag"` 시 노출 기록도 생략)
- backend의 D1 환경변수(`CLOUDFLARE_ACCOUNT_ID` / `CLOUDFLARE_API_TOKEN` / `D1_DATABASE_ID`) 누락 시 **모든 쿼리가 조용히 빈 결과 반환** — 콘솔에 데이터 안 보일 때 첫 의심 지점
- 콘솔 UI 수정 시 KO/EN 동시 반영(AGENTS.md §5), shadcn/ui 컨벤션(§4) 유지
- demo-app 자체는 plain Tailwind/KO 단일 — 의도된 격리(설계서 `20260517_arch.md` §9.1). 콘솔 작업과 혼동 금지

**검증 방법** (작업 완료 후):
1. `bash examples/demo-app/scripts/seed.sh` 1회 실행 → 5개 호출 모두 201 또는 409
2. `docker compose -f docker-compose.dev.yml up`
3. 시크릿 창 여러 개로 demo-app(8082) 방문 + 카드/배너 클릭 누적
4. 콘솔(8081) Feature Flags 페이지에서 `home_layout_v1` 카드에 `control N / list M` 두 칩 표시 확인 (이슈 1)
5. 콘솔 "실험 관리" → `home_layout_exp_v1` 진입 → P(treatment>control), uplift 숫자 표시 확인 (이슈 2)
