## 0. 배경

- 현재 실험 플랫폼(`backend/` + `frontend/`)은 관리자 콘솔 + API로 구성됨
- 실제 외부 앱이 플랫폼을 어떻게 활용하는지 검증할 **데모 사이트**가 필요
- 광고/스폰서 표시 최적화 시나리오에 관심 있음 (pseudo-lab.com 컨텍스트)

## 1. 플랫폼이 외부 앱에 제공하는 API 표면

| API | 용도 | 호출 시점 |
|-----|------|----------|
| `GET /api/v1/feature-flags/decide` | 분기 결정(`control`/`treatment`) | 페이지 렌더 |
| `GET /api/v1/experiments/{id}/assign/{user_id}` | 통계 분석용 실험 배정 | 페이지 로드 1회 |
| `POST /api/v1/capture` | 전환/행동 이벤트 보고 | 사용자 액션 |
| `POST /api/v1/identify` | 사용자 메타데이터 등록 | 로그인/식별 |

> ⚠️ **경로 함정**: events 라우터는 `backend/app/api/v1/api.py:12`에서 `prefix=""`로 마운트돼 있어 `/events/` 네임스페이스가 없습니다. 다른 라우터(`/feature-flags`, `/experiments` 등)와 패턴이 다르므로 `/api/v1/events/capture`로 적으면 404가 납니다.

핵심 동작:
- **Feature Flag decide**: 규칙(priority 순) → 세그먼트 매칭 → 시간 윈도우 → CRC32 해시 rollout. 같은 user_id는 항상 같은 결과 (sticky)
- **Experiment 결과**: assignments × event_log(primary_metric) JOIN → Beta 샘플링 1만 번으로 `P(treatment > control)`, 카이제곱 SRM 경고

## 2. 시나리오 후보

### A. 광고/스폰서 시나리오

#### A-1. 스폰서 배너 슬롯 최적화 ⭐ 데모용 1순위
| 항목 | 내용 |
|------|------|
| 가설 | in-feed 배치가 사이드바보다 CTR이 높을 것 |
| Variants | `control`: 우측 사이드바 / `treatment`: 콘텐츠 피드 인라인 |
| primary_metric | `sponsor_clicked` |
| 세그먼트 활용 | "비로그인 방문자", "특정 cohort" 등 분리 |
| 시간 윈도우 | 캠페인 기간 `starts_at`/`ends_at`으로 자동 종료 |

→ Feature Flag 규칙의 모든 축(세그먼트 + rollout% + 시간 윈도우)을 자연스럽게 사용

#### A-2. 광고 카피 다국어/문구 테스트
- Variants: "자세히 알아보기" / "지금 신청하기" / "무료 체험 →"
- primary_metric: `sponsor_clicked` + `sponsor_landed`
- AGENTS.md의 KO/EN i18n 규칙과 자연스럽게 연결

#### A-3. 광고 빈도/위치 피로도 테스트
- Variants: 페이지당 1회 노출 / 무제한 노출
- primary_metric: `sponsor_clicked`, `session_bounced`(부정 지표)
- 단순 CTR 외에 이탈률 같은 부작용도 같이 측정

### B. 커뮤니티/스터디 모집 시나리오

#### B-1. 스터디 카드 레이아웃 (가장 쉬운 시작점)
- Variants: 그리드 3열 / 리스트 1열
- primary_metric: `study_card_clicked` → `study_apply_started`
- SDK·이벤트 수집을 익히는 워밍업으로 적합

#### B-2. 지원서 폼 길이 (퍼널 테스트)
- Variants: 한 페이지 5필드 / 3단계 분할
- primary_metric: `apply_completed`
- 펀넬 분석 API(`/api/v1/analytics/funnels`)와 가장 잘 맞음

#### B-3. Discord 초대 위치
- Variants: 헤더 고정 / 푸터 / 스크롤 슬라이드인
- primary_metric: `discord_invite_clicked`

### C. 온보딩/리텐션 시나리오 (장기 학습용)

#### C-1. 첫 방문자 환영 모달
- Variants: 모달 / 비표시 / 인라인 배너
- primary_metric: `return_visit_7d` (단순 클릭이 아닌 리텐션)
- `/api/v1/analytics/retention` 활용

## 3. 추천: "가짜 PseudoLab" 데모 사이트 (A-1 + B-1 묶음)

한 데모 사이트에 시나리오 두 개를 동시 운영:

```
demo-pseudo-lab/
├── 홈                  ← B-1: 스터디 카드 레이아웃 (그리드/리스트)
├── 스터디 목록         ← A-1: 스폰서 배너 슬롯 (사이드/인라인)
├── 스터디 상세         ← (전환 지점)
└── 지원 폼             ← (전환 완료)
```

이유:
1. B-1은 UI만 다른 단순 케이스 — SDK·이벤트 수집을 익히는 워밍업
2. A-1은 세그먼트·시간 윈도우·노출 분석을 다 끌어쓰는 본격 케이스
3. 한 사이트에 동시 진행 → "여러 실험 병행 운영" 현실 시뮬레이션
4. 광고주에게 시연하기에도 매력적

## 4. 광고 가치 측정(advanced) — 비전 연계

PseudoLab 광고에서 진짜 중요한 건 단순 클릭이 아니라 "우리 멤버가 광고주에게 가치 있는 트래픽인가":
- 광고 노출 시 cohort/role 메타데이터를 함께 기록 (identify API)
- 노출 요약을 cohort별로 분해 → "Builder 코호트 CTR X%, Student Y%"
- 단순 PV가 아니라 분류된 인사이트로 광고주 가치 증명

→ 프로젝트 비전("커뮤니티 데이터의 실제 가치 측정")과 일치

```
├── frontend/         (관리자 콘솔, 그대로)
└── examples/

        │   ├── pages/

```

독립 앱으로 두는 이유: 관리자 콘솔과 "실제 사용자가 쓰는 앱"의 책임을 분리해야 플랫폼을 외부 SDK 사용자 관점에서 실전 테스트 가능.

### 최소 SDK 골격

```ts
// examples/demo-pseudo-lab/src/lib/experiment-sdk.ts
const API = import.meta.env.VITE_PLATFORM_API; // http://localhost:8000/api/v1

function getUserId() {
  let id = localStorage.getItem('demo_uid');
  if (!id) { id = crypto.randomUUID(); localStorage.setItem('demo_uid', id); }
  return id;
}

export async function decideFlag(flagKey: string): Promise<'control'|'treatment'> {
  const uid = getUserId();
  const res = await fetch(`${API}/feature-flags/decide?flag_key=${flagKey}&user_id=${uid}`);
  return (await res.json()).data.variant;
}

export async function track(eventName: string, properties?: Record<string, any>) {
  await fetch(`${API}/capture`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ user_id: getUserId(), event_name: eventName, properties }),
  });
}

export async function identify(meta: { cohort_id?: string; team_name?: string; role?: string }) {
  await fetch(`${API}/identify`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ user_id: getUserId(), ...meta }),
  });
}
```

```tsx
// hooks/useFlag.ts
export function useFlag(flagKey: string) {
  const [variant, setVariant] = useState<'control'|'treatment'|null>(null);
  useEffect(() => { decideFlag(flagKey).then(setVariant); }, [flagKey]);
  return variant;
}

// pages/HomePage.tsx 예시
function HomePage() {
  const variant = useFlag('home_cta_v1');
  if (!variant) return <Skeleton />;
  return (
    <button onClick={() => track('cta_clicked', { variant })}>
      {variant === 'treatment' ? '지금 시작' : '구매하기'}
    </button>
  );
}
```

## 6. 구축 단계

1. **준비** — 관리자 콘솔에서 Flag(`home_layout_v1`, `sponsor_slot_v1`)와 실험(primary_metric=`sponsor_clicked` 등) 사전 생성
2. **SDK 작성** — 위 50줄 SDK로 시작
3. **첫 페이지** — 홈에 B-1 시나리오 적용. `docker-compose.dev.yml`에 `demo-pseudo-lab` 서비스 추가 (포트 8082)
4. **노출/전환 검증** — 관리자 콘솔 Feature Flag 노출 요약, 실험 결과(uplift/probability) 확인
5. **시나리오 확장** — A-1 추가, 세그먼트 타게팅, 시간 윈도우, rollout 5%→50%→100% 점진 확장

## 7. 주의 사항

- **CORS는 `["*"]`** (`backend/app/core/config.py:11`) — 로컬은 OK지만 운영은 origin 명시 필요
- **D1 환경변수 필수** (`backend/app/db/d1.py:9-30`) — `CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_API_TOKEN`, `D1_DATABASE_ID` 없으면 모든 쿼리가 조용히 빈 결과 반환
- **Decision API에 캐싱 없음** — 매 페이지 로드마다 호출되면 latency 부담. SDK 단에서 sessionStorage 캐시 권장
- **Identify는 upsert** — cohort_id 누락 시 기존값을 NULL로 덮어씀. SDK에서 부분 업데이트 패턴 주의
