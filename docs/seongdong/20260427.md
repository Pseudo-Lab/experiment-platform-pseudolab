## 2. 실험 관리

### 2-1. 실험 생성
1. 사이드바 → **실험(A/B)** 클릭
2. **새 실험 생성** 버튼 클릭
3. 아래 값으로 입력:
   - 이름: `테스트 실험 2026-04`
   - 가설: `새 UI가 주간 출석률을 높일 것이다`
   - Primary Metric: `weekly_session_attended`
   - Variants: `control` (50%) / `treatment` (50%)
4. 저장

**체크포인트**
- 실험 목록에 방금 만든 실험이 나타나는지
- 상태가 `draft`인지
- 상세 페이지 진입 시 variants가 2개 표시되는지

### 2-2. traffic_ratio 합계 오류 케이스
1. 새 실험 생성 시 control 30% / treatment 30% 로 저장 시도
2. **기대**: 저장 안 되고 에러 메시지 표시 ("traffic_ratio 합계는 1.0이어야 합니다")

### 2-3. 상태 전환 — draft → running
1. 실험 상세 페이지 진입
2. **"running 시작"** 버튼 클릭
3. **체크포인트**: 상태가 `running`으로 바뀌고, draft 전환 버튼은 사라지는지

### 2-4. 상태 전환 — running → paused → running
1. running 상태에서 **"일시정지"** 클릭 → `paused` 확인
2. paused 상태에서 **"재시작"** 클릭 → `running` 확인

### 2-5. 상태 전환 — running → completed
1. running 상태에서 **"완료"** 클릭 → `completed` 확인
2. **체크포인트**: completed 상태에서는 어떤 상태 전환 버튼도 없어야 함

## 3. 유저 할당 + 이벤트 데이터 심기 (API 직접 호출)

Analytics와 결과 분석은 데이터가 있어야 의미 있는 화면이 나온다.  
아래 curl을 로컬 터미널에서 실행해서 데이터를 심는다.

> `{EXP_ID}` = 2-1에서 만든 실험의 ID (실험 상세 URL에서 확인 가능)  
> API Base: 운영 백엔드 URL (프론트 URL에서 포트/경로 확인 필요)

```bash
# 실험 ID 확인 (실험 목록 조회)
curl https://exp.pseudolab-devfactory.com/api/v1/experiments

# 상태를 running으로 변경 (아직 안 했다면)
curl -X PATCH https://exp.pseudolab-devfactory.com/api/v1/experiments/{EXP_ID} \
  -H "Content-Type: application/json" \
  -d '{"status": "running"}'

# 유저 5명 할당
for user in user-001 user-002 user-003 user-004 user-005; do
  curl "https://exp.pseudolab-devfactory.com/api/v1/experiments/{EXP_ID}/assign/$user"
done

# 사용자 식별
for user in user-001 user-002 user-003 user-004 user-005; do
  curl -X POST https://exp.pseudolab-devfactory.com/api/v1/identify \
    -H "Content-Type: application/json" \
    -d "{\"user_id\": \"$user\", \"cohort_id\": \"cohort-2026\", \"cohort_name\": \"2026 1기\", \"team_name\": \"팀A\", \"role\": \"member\"}"
done

# 이벤트 수집 — treatment 유저 3명 + control 유저 1명에게 전환 이벤트
for user in user-001 user-002 user-003; do
  curl -X POST https://exp.pseudolab-devfactory.com/api/v1/capture \
    -H "Content-Type: application/json" \
    -d "{\"user_id\": \"$user\", \"event_name\": \"weekly_session_attended\", \"properties\": {\"week_number\": 1}}"
done

curl -X POST https://exp.pseudolab-devfactory.com/api/v1/capture \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user-004", "event_name": "weekly_session_attended", "properties": {"week_number": 1}}'

# 추가 이벤트 (Analytics 트렌드·퍼널용)
curl -X POST https://exp.pseudolab-devfactory.com/api/v1/capture \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user-001", "event_name": "deliverable_submitted", "properties": {"type": "blog"}}'
```

## 4. 실험 결과 확인 (UI)

1. 실험 상세 페이지 → **"결과 조회"** 버튼 클릭
2. **체크포인트**:
   - control / treatment 각각 users, conversions, rate 표시되는지
   - `probability_treatment_wins` 값이 0~1 사이 float인지
   - `uplift` 값이 표시되는지
   - `srm_warning` 표시 여부

---

## 5. 의사결정 기록 (UI)

1. 실험 상세 → **결정 기록** 섹션
2. `SHIP` 선택 → 이유 입력 → 결정자 입력 → 저장
3. **체크포인트**: 이력 목록에 방금 기록이 나타나는지
4. 같은 실험에 `HOLD`로 한 번 더 저장 → 이력이 2개 쌓이는지 (수정이 아닌 누적)

---

## 6. 학습 노트 (UI)

1. 실험 상세 → **학습 노트** 섹션
2. 내용 입력 → 저장
3. **체크포인트**: 목록에 표시되는지

---

## 7. 회고 가이드 설정 (UI)

1. 실험 상세 → **회고 가이드 설정** 섹션
2. 노출 시작일: 오늘 날짜, 노출 기간: 7일 입력 → 저장
3. **체크포인트**: "YYYY-MM-DD 부터 7일간 노출" 문구 표시되는지
4. **회고 작성 현황** 표시 여부 확인 (마이그레이션 007 적용 시)

## 8. Feature Flags (UI)

1. 사이드바 → **Feature Flags** 클릭
2. **새 Flag 생성** 클릭:
   - flag_key: `magic-week-v1`
   - 설명: `새 주간 미션 UI`
   - 롤아웃: 50%
   - 활성화: ON
3. **체크포인트**: 목록에 나타나는지, 상태 토글 작동하는지

### 결정론적 배정 확인 (API)
```bash
# 같은 user + flag_key는 항상 같은 결과
curl "https://exp.pseudolab-devfactory.com/api/v1/feature-flags/decide?flag_key=magic-week-v1&user_id=user-001"
curl "https://exp.pseudolab-devfactory.com/api/v1/feature-flags/decide?flag_key=magic-week-v1&user_id=user-001"
# 두 번 다 동일한 variant 반환되어야 함
```

### 비활성화 시 전원 control 확인 (UI + API)
1. UI에서 `magic-week-v1` 토글 → OFF
2. `curl "https://exp.pseudolab-devfactory.com/api/v1/feature-flags/decide?flag_key=magic-week-v1&user_id=user-001"`
3. **기대**: `"variant": "control"` (enabled=false면 항상 control)

## 9. Analytics 대시보드 (UI)

> 3단계에서 이벤트 데이터를 심은 후 테스트

1. 사이드바 → **Analytics > 이벤트** 탭
   - 이벤트 목록 테이블에 `weekly_session_attended` 로그 보이는지
   - 이벤트 이름으로 필터링 작동하는지

2. **Analytics > 트렌드** 탭
   - 이벤트 선택 드롭다운에 `weekly_session_attended` 있는지
   - 날짜 범위 설정 후 꺾은선 그래프 렌더링되는지

3. **Analytics > 퍼널** 탭
   - 스텝 1: `weekly_session_attended`, 스텝 2: `deliverable_submitted` 입력
   - "분석" 클릭 → 막대 그래프 + 전환율 표시되는지

4. **Analytics > 리텐션** 탭
   - `weekly_session_attended` 선택
   - 히트맵 테이블 렌더링되는지 (데이터가 적으면 셀이 sparse할 수 있음)

## 10. 회고 제출 (API — 페이지 미구현 상태)

회고 가이드 페이지(`/reflection/:experimentId`)는 Lovable이 아직 개발 전.  
데이터 흐름만 API로 확인.

```bash
# 제출 여부 확인
curl "https://exp.pseudolab-devfactory.com/api/v1/reflections/check?user_id=user-001&experiment_id={EXP_ID}"
# 기대: {"submitted": false}

# 회고 제출
curl -X POST https://exp.pseudolab-devfactory.com/api/v1/reflections \
  -H "Content-Type: application/json" \
  -d '{
    "experiment_id": "{EXP_ID}",
    "user_id": "user-001",
    "project_id": "project-001",
    "project_type": "implementation",
    "output_types": ["github", "post"],
    "response_good": "매주 꾸준히 모였어요",
    "response_blocked": "데이터 처리가 느렸어요",
    "response_goal": "GitHub README 완성",
    "final_output_type": "github"
  }'
# 기대: 201

# 중복 제출 시도
# 기대: 409 "이미 이 실험에 회고를 제출했습니다"

# 실험 상세 > 회고 현황 섹션에서 total_completed=1 표시 확인
```

```
① https://exp.pseudolab-devfactory.com/dashboard 접속

⑥ 실험 상세 > 학습 노트
⑦ 실험 상세 > 회고 가이드 설정
⑧ Feature Flags — 생성, 토글, 비활성화 시 control 확인
⑨ Analytics — 이벤트/트렌드/퍼널/리텐션 탭
⑩ [API] 회고 제출 + 중복 방지 확인
```
